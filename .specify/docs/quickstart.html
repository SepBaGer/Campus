5-Minute Quickstart

Go from zero to a fully tracked specification project with ambient intelligence in five steps.

1

Initialize Your Project

> /sdd:init ╔══════════════════════════════════════════════════════╗ ║ SDD — Spec Driven Development ║ ║ by metodologIA ║ ╚══════════════════════════════════════════════════════╝ Step 1: Running upstream IIC/kit init... What happened

SDD created the .specify/ directory with pipeline state, initialized the heartbeat sentinel, and detected that governance files are missing. It tells you exactly what to run next.

2

Define Governance

terminal
> /sdd:00-constitution User: "I'm building a fintech API. Security-first, TDD mandatory, REST API with OpenAPI spec, PostgreSQL database."
What happened

SDD creates CONSTITUTION.md with principles derived from your description. It maps "security-first" to input sanitization rules, "TDD mandatory" to test-before-code requirements. Every principle gets a traceable ID (P-I, P-II, ...).

📜 CONSTITUTION.md preview

# Project Constitution ## Principles P-I. Security First All inputs sanitized. OWASP Top 10 compliance mandatory. Auth via JWT with refresh rotation. P-II. Test-Driven Development Tests written before implementation. ≥80% coverage gate. BDD Gherkin for acceptance criteria. P-III. API-First Design OpenAPI 3.1 spec precedes implementation. Breaking changes require version bump. P-IV. Data Integrity PostgreSQL with migrations. No raw SQL in app layer. Audit trail on all financial operations.
3

Specify Your First Feature

terminal
> /sdd:spec User: "Create a feature for user authentication with email/password, OAuth2 Google, and MFA via TOTP."
What happened

SDD generates specs/001-user-auth/spec.md with FR-001 to FR-005 (functional requirements), US-001 to US-003 (user stories), and SC-001 to SC-003 (success criteria). Each requirement is traceable back to your constitution principles.

📜 specs/001-user-auth/spec.md preview

# Feature: User Authentication ## Functional Requirements FR-001 Email/password registration with bcrypt hashing FR-002 Email/password login returning JWT access + refresh tokens FR-003 OAuth2 Google sign-in with account linking FR-004 TOTP-based MFA enrollment and verification FR-005 Password reset via time-limited email token ## User Stories US-001 As a user, I can register with my email so I have an account US-002 As a user, I can sign in with Google for faster access US-003 As a user, I can enable MFA to secure my account ## Success Criteria SC-001 Login <200ms p95 latency SC-002 Failed login locks after 5 attempts for 15 min SC-003 100% of auth endpoints covered by integration tests
4

Plan, Verify, and Test

terminal
> /sdd:plan ✓ Technical design document created ✓ Architecture decisions recorded ✓ Dependency graph generated GATE 1 PASSED — plan is consistent with spec > /sdd:check ✓ Quality checklist generated (14 items) ✓ Security items: 4 | Performance items: 3 | Coverage items: 7 > /sdd:test ✓ BDD Gherkin scenarios generated ✓ 8 scenarios across 3 feature files ✓ Assertion hashes locked: TS-001 sha256:a3f2... TS-002 sha256:9c1b... TS-003 sha256:d4e7...
What happened

Three phases executed in sequence. The plan created a technical design and passed Gate 1. The checklist generated quality verification items. The test phase produced BDD Gherkin scenarios with cryptographic assertion hashes — if anyone changes a test without updating the hash, SDD detects the tampering.
