spec.md Phase 1 
# Feature: User Authentication

## Requirements
- FR-001: Register with email/password
- FR-002: Login returns JWT (1h expiry, RS256)
- FR-003: OAuth2 Google sign-in (PKCE flow)
- FR-004: MFA via TOTP (RFC 6238)
- FR-005: Password reset via email token

## User Stories
- US-001: As a user, I want to register
  with my email so I can access the platform
- US-002: As a user, I want to enable MFA
  so my account is more secure
- US-003: As a user, I want to sign in
  with Google for faster onboarding

## Success Criteria
- Login latency < 200ms (p95)
- MFA setup completes in < 60 seconds
- Password reset token valid for 15 min
plan.md Phase 2 
# Technical Plan: User Authentication

## Architecture
Provider: Supabase Auth + custom JWT
Pattern: Ports & Adapters
Edge: Vercel Middleware (token verify)

## Data Model
table users
  id         uuid PRIMARY KEY
  email      text UNIQUE NOT NULL
  mfa_secret text ENCRYPTED
  created_at timestamptz DEFAULT now()

## API Endpoints
POST /api/auth/register  → FR-001
POST /api/auth/login     → FR-002
GET  /api/auth/callback  → FR-003
POST /api/auth/mfa/setup → FR-004
POST /api/auth/reset     → FR-005

## RLS Policies
Users can only read/update own row.
Service role bypasses for admin ops.

# RAG Memory: API Specification

## Abstract
OpenAPI 3.1 specification for the FinTech
platform REST API. Defines 23 endpoints
across 4 resource groups: auth, users,
transactions, and admin.

## Key Takeaways
- Bearer token auth on all endpoints
- Rate limit headers: X-RateLimit-*
- Pagination via cursor, not offset
- Webhook callbacks for async operations

## Relevant Insights
The /transactions endpoint requires
idempotency keys (Idempotency-Key header).
Error responses follow RFC 7807.
